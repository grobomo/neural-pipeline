# Scope Phase Reference

## Purpose
Define exactly WHAT we are doing. Draw the boundaries. Write acceptance
criteria. Clarify what's in scope and what's explicitly out of scope.

## Manager Role
- Read the Why phase output for motivation and success definition
- Define clear boundaries: what's included and what's excluded
- Write acceptance criteria that the Verify phase can check
- Identify dependencies and prerequisites

## Worker Instructions Template
Given the motivation and success definition from the Why phase:
1. What specific deliverables are expected?
2. What is explicitly out of scope?
3. What are the acceptance criteria? (specific, testable)
4. What dependencies or prerequisites exist?
5. What assumptions are we making?

## Success Criteria Examples
- "At least 3 specific acceptance criteria defined"
- "Out-of-scope items explicitly listed"
- "All assumptions documented"
- "Dependencies identified with resolution paths"

## Common Pitfalls
- Scope creep: adding features the user didn't ask for
- Vague criteria: "works well" instead of "returns 200 OK with JSON body"
- Missing edge cases that will cause Verify phase failures
