# Why Phase Reference

## Purpose
Determine WHY this task matters. What pain point does it solve? What goal
does it serve? What does success look like for the user?

## Manager Role
- Extract the core motivation from the user request
- Identify the underlying pain point (not just the surface ask)
- Define what "success" means in the user's terms
- Check memory for similar past tasks and their outcomes

## Worker Instructions Template
Given the user's request, analyze:
1. What problem is the user trying to solve?
2. Why does this matter to them?
3. What would a successful outcome look like?
4. Are there any implicit requirements not stated explicitly?

## Success Criteria Examples
- "Motivation clearly stated in 1-2 sentences"
- "Success definition is measurable, not vague"
- "Implicit requirements surfaced and documented"

## Common Pitfalls
- Taking the request too literally (user says "fix the button" but the real
  problem is the whole form UX)
- Not asking "why" enough -- the first answer is rarely the deepest
- Overcomplicating the motivation analysis for simple tasks
